Why Workflow Wins

为什么"让 AI 直接写"行不通？

纯 AI 生成代码看似美好，实则暗藏致命陷阱。AutoForge 采用工作流驱动的方式， 从根本上解决了这些问题。

纯 AI "直接写"

Copilot / ChatGPT / 通用大模型
  • 幻觉问题 (Hallucination) AI 编造不存在的 API 名称、虚构参数签名，编译直接报错
  • 风格不一致 每次生成的命名风格、异常处理、日志格式都可能不同
  • 上下文窗口有限 无法同时理解整个项目架构，生成的代码往往不符合既有模式
  • 无法调用真实工具链 不能编译验证、不能跑测试、不能分析硬件拓扑
  • 不可控 & 不可追溯 黑盒生成，无法解释为什么这么写，出了问题难以定位
VS

AutoForge 工作流

结构化 Agent + 确定性模板 + 人工审核
  • 零幻觉：Skill + 模板兜底 API 签名来自解析后的 DLL 元数据，不靠 AI 记忆；确定性模板保证框架调用正确
  • 100% 风格统一 所有代码由公司标准模板渲染，AI 只填充业务差异部分，风格天然统一
  • 分阶段工作流，逐步深入 每个 Agent 只处理一个明确子任务，完美切分上下文边界
  • 闭环验证 每一阶段输出都经过编译验证和沙盒测试，失败自动回滚重试
  • 全链路透明 & 可审计 每一步决策有日志、有上下文、有 diff，满足工业审计要求
核心区别：AutoForge 不是"让 AI 自由发挥"
我们把 AI 置于一条严格受控的工业流水线上。每个 Agent 的输入、输出、 约束条件都是确定的。AI 只在被允许的空间内发挥创造力，其余部分全部由 确定性模板和编译器保证正确性。这就像让 AI 在"护栏"里跑， 而不是在旷野上自由奔跑。

Under the Hood

技术原理一览

AutoForge 的核心并非"一个大模型"，而是一套协作式 Agent 工作流 + 确定性代码渲染引擎。

输入层
I/O 点位表 (Excel)
硬件 BOM 清单
MES API 文档
自然语言描述
核心引擎
Skill 匹配引擎
通过解析 DLL / NuGet 包的反射元数据，构建 skill.json 描述文件。 Agent 按硬件类型查表匹配可用 Skill，消除 AI "幻觉" API 调用。
确定性模板引擎
MainWindow、Service、ViewModel 等结构由 Scriban 模板渲染。 框架调用代码 100% 确定生成，杜绝 AI 篡改框架层级接口。
多阶段 Agent 编排
每个 Agent 只处理一个子任务（如 PLC 驱动、视觉流程、I/O 映射）。 编排器按依赖关系调度，前一阶段的输出是后一阶段的输入。
沙盒编译验证
每一阶段生成的代码都在隔离沙盒中执行 dotnet build。 编译失败自动携带错误上下文回传 Agent 进行修复，最多 N 轮重试。
输出层
完整 .sln 项目
编译通过的可执行文件
变更日志 & Diff
合规审计报告
01
反射式 Skill 发现
通过 .NET Reflection 提取 DLL 中的类名、方法签名、参数类型，自动构建结构化的 Skill 描述文件。 LLM 在生成代码时严格引用这些真实签名，而非从训练数据中"回忆"。
02
模板 + AI 混合渲染
结构性代码（MVVM 骨架、DI 注册、界面布局）由 Scriban 模板确定性生成。 仅业务逻辑层（工艺流程、异常处理策略）交由 LLM 在约束条件下填充。 两者的边界由清晰的接口契约定义。
03
编译驱动的自修复循环
每一步输出不是"看起来正确的代码"，而是"通过 dotnet build 验证的代码"。 编译器错误信息会自动回传到 Agent 的上下文中，驱动 AI 自主修复， 直到 0 Error 达成或超过重试阈值。
